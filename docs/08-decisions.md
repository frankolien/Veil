# Veil — Architecture Decision Records

Status: draft · Last updated: 2026-05-29

Lightweight ADRs in the Michael Nygard format: **Context · Decision · Consequences · Status**.

Each ADR is the size it needs to be — not padded. New decisions get appended; superseded ones get a `Status: Superseded by ADR-NNN` line, not deleted.

---

## ADR-001 — `NUM_TICKS = 4` for v1

**Context.** Per-order gas in `placeOrder` is `O(NUM_TICKS)` because the per-tick aggregation loop must be unrolled at compile time (FHEVM cannot run encrypted loop bounds). A larger tick grid gives finer price granularity but multiplies the gas cost of every order.

**Decision.** Fix `NUM_TICKS = 4` for v1.

**Consequences.** Four-tick granularity is too coarse for a real market. v1 is a primitive demonstration, not a production venue. v2 will parameterise `NUM_TICKS` once the Week 5 gas profile tells us how high we can go without `placeOrder` exceeding Sepolia's block budget.

**Status.** Accepted for v1. Re-opened for v2 (`05-roadmap.md` Week 5).

## ADR-002 — Plaintext-bps pro-rata, not encrypted division

**Context.** Pro-rata at the marginal tick requires `size · α` where `α ∈ [0, 1]`. The natural representation is `FHE.div(FHE.mul(size, alphaNum), alphaDen)`, but FHEVM blocks division by encrypted divisors (see `02-algorithm.md`).

**Decision.** Submit `marginalBuyBps` and `marginalSellBps` as plaintext `uint16` values in `[0, 10_000]`. Compute `fill = FHE.div(FHE.mul(size, marginalBps), 10_000)`. Both `marginalBps` and `10_000` are plaintext, so `FHE.div` is legal.

**Consequences.** Pro-rata granularity is bps (1 part in 10,000), which is finer than any reasonable market needs. The solver rounds down, so aggregate fills can be strictly less than `M(c*)` — intentional dust. We accept publishing `marginalBuyBps`/`marginalSellBps` in plaintext; this is one bit of public information per side per batch, which is the minimum honest summary of pro-rata.

**Status.** Accepted.

## ADR-003 — Off-chain solver, publicly-detectable correctness

**Context.** Computing the clearing tick and pro-rata bps from publicly-decrypted aggregates is cheap off-chain and expensive on-chain (it requires iterating ticks, summing, picking max, computing bps — none under FHE, but enough to add meaningful gas). Either anyone re-runs the computation off-chain to verify (cheap, requires social challenge), or the contract re-runs it (gas-expensive, fully trustless).

**Decision.** v1 ships with off-chain `computeClearing` (`contracts/tasks/Veil.ts`). `submitClearing` accepts any valid-shaped `(uint8, uint16, uint16)` tuple from any caller and writes it to state without re-verification.

**Consequences.**
- Trust shape: anyone can re-run `computeClearing` on the publicly-decryptable aggregates and detect a wrong submission. Detection is seconds; on-chain remediation is "submit the correct values" — but `submitClearing` can only run once per batch (`BatchAlreadyCleared`).
- The race risk: a malicious solver can submit wrong values first; honest parties have no on-chain replay. v1 documents this as a known limitation.
- v2 should add either on-chain re-verification (gas cost: ~doubles `submitClearing`) or a challenge window (latency cost: batches stay in flux for an extra window).

**Status.** Accepted for v1. **Re-opened for v2 — decision criteria pinned in `07-security.md` Threat 4.**

## ADR-004 — `viaIR: true` on the contracts compiler

**Context.** `submitClearing`'s per-order body uses ~15 local FHE handles per loop iteration (tickEq, tickGt, tickLt, isBuyAbove, isSellBelow, fullFill, isBuyAt, isSellAt, buyMarginal, sellMarginal, marginalFill, …). The default Solidity codegen runs out of stack slots ("stack too deep").

**Decision.** Enable `viaIR: true` in `contracts/hardhat.config.ts`. Compile via Yul; the IR codegen allocates registers more flexibly and handles the deep stack.

**Consequences.**
- Compilation is ~3× slower (acceptable for a Hardhat project).
- Generated bytecode is slightly different from the default codegen; this is a documented Solidity feature, not a footgun, but worth knowing if we ever read disassembly.
- Without `viaIR` we would either need to refactor every FHE loop body into a helper function (extra calldata copies for handles, more gas) or shrink the loop. Refactoring is the alternative; we prefer the compiler flag.

**Status.** Accepted.

## ADR-005 — `batchBlocks = 10` for the prototype

**Context.** The constructor takes `batchBlocks` to size the auction window. Short windows give a tight feedback loop for demoing; long windows are realistic for a production venue with thin order flow.

**Decision.** Deploy with `batchBlocks = 10` (~2 minutes on Sepolia at 12s blocks).

**Consequences.**
- Demos take 2 minutes per batch — short enough for the 3-minute pitch.
- Real-world deployment would want 5–30 minute batches (60–300 blocks) to amortise the per-batch overhead and give traders time to construct orders.
- The contract parameter is `immutable`; changing it requires redeploying. The PRD's Phase A success metric ("≥ 5 batches cleared end-to-end") was sized against `batchBlocks = 10`.

**Status.** Accepted for v1. Production deployment picks a higher value.

## ADR-006 — Next.js Webpack mode, not Turbopack

**Context.** Next.js 16 defaults to Turbopack for both `dev` and `build`. The `@zama-fhe/sdk` bundle contains a right-associative `**` exponentiation operator that Turbopack's parser rejects (a known issue at SDK version 0.4.x).

**Decision.** Both `web/` and `tokenops/` run `next dev --webpack` and `next build --webpack`.

**Consequences.**
- Slower dev-server startup (~2s extra cold) and slightly slower builds. Tolerable.
- We diverge from the Next.js 16 default; new contributors need to know this. Documented in `04-stack.md`.
- If Turbopack adds full ES2022 support upstream (or if the SDK changes its bundling), this ADR can be deprecated.

**Status.** Accepted, contingent on Turbopack parser.

## ADR-007 — Local `WagmiSigner` shim instead of `@zama-fhe/react-sdk/wagmi`

**Context.** `@zama-fhe/react-sdk/wagmi`'s `WagmiSigner` imports `watchConnection` from `wagmi/actions`. Current wagmi exports `watchConnections` (plural). Importing `WagmiSigner` fails at module-load time.

**Decision.** Both web apps include a local `lib/zama-signer.ts` that implements the `GenericSigner` interface against `wagmi/actions` directly. We omit the optional `subscribe` lifecycle hook (the only thing that depended on `watchConnection`).

**Consequences.**
- We carry a small piece of upstream code in our repo, in two copies. Acceptable; the shim is ~30 lines.
- If `@zama-fhe/react-sdk` ships a fix we can drop the shim by deleting two files and changing the import.
- The shim is documented in its own file header.

**Status.** Accepted, contingent on upstream fix.

## ADR-008 — 4-tick on-chain grid, 7-tick mock visual on the landing page

**Context.** The trade app needs to render the actual on-chain tick grid (`NUM_TICKS = 4`). The landing page's hero orderbook is a vibe demo with animated bids/asks — it benefits from more rows for visual rhythm.

**Decision.** `web/lib/use-veil-lifecycle.ts` returns a 4-tick `Book` for `/app`. `web/components/veil/orderbook.tsx`'s `makeBook(seed)` generates a 7-tick mock `Book` used only on the landing page.

**Consequences.**
- Two shapes of `Book` floating around the codebase. Mitigated by `useVeilLifecycle` being the only producer of the on-chain shape and `makeBook` being the only producer of the mock shape.
- The mock visual is honest fiction: clearly an animation, not a representation of live state.
- When `NUM_TICKS` is parameterised in v2, the on-chain shape will inherit the new value automatically; the mock visual is independent.

**Status.** Accepted for v1.

## ADR-009 — Pin `placeOrder` gas to 15M instead of `eth_estimateGas`

**Context.** viem's `useWriteContract` auto-runs `eth_estimateGas` before submitting. For FHE-heavy calls, the wallet's RPC returns an estimate that exceeds Sepolia's block gas limit (~30M) and refuses to forward the tx with "gas limit too high". Actual `placeOrder` cost is well under 15M.

**Decision.** Pass `gas: 15_000_000n` explicitly in the `writeContractAsync` call inside `OrderTicket.submit` (`web/components/veil/trade-app.tsx`).

**Consequences.**
- The wallet UI shows a 15M gas cap, which can look alarming. Mitigated by the in-form copy ("encrypts to … sealed on-chain").
- If actual gas usage exceeds 15M after a contract change, the tx will out-of-gas revert and the receipt-wait path will surface it.
- The Week 5 gas profile produces a real number; this ADR is revisited then with the measured budget plus headroom.

**Status.** Accepted, pending measurement.

## ADR-010 — ERC-7984 operator pattern over per-order encrypted approvals

**Context.** ERC-7984 has no per-amount allowance like ERC-20. It has an *operator model*: `setOperator(address operator, uint48 until)` grants the operator full transfer rights until the expiry. Two options for collateral escrow inside `placeOrder`:

1. User signs a per-order encrypted approval with input proof, contract calls `confidentialTransferFrom(user, contract, encryptedAmount, inputProof)`.
2. User calls `setOperator(VeilBatchAuction, +24h)` once, then every `placeOrder` calls `confidentialTransferFrom(user, contract, alreadyAllowedHandle)` with no proof.

**Decision.** v2 ships option 2 (one-time operator grant). The frontend renders a single "Approve Veil to spend cWETH/cUSDC for 24h" step before the first order; subsequent orders need only the encrypt + placeOrder flow.

**Consequences.**
- Once approved, Veil can pull *any* amount of the operator-allowed token from the user's account until the expiry. This is the standard ERC-7984 trust assumption; we document it.
- UX: one extra signature on first use, zero overhead thereafter. The alternative (option 1) requires an extra encrypted input proof per `placeOrder` — ~doubles client-side encryption cost.
- The 24h expiry is a parameter; longer = better UX, shorter = better security. We default to 24h and let the user re-approve.

**Status.** Proposed (design), to be accepted on first v2 deployment. Implementation tracked in `05-roadmap.md` Week 2/3.

## ADR-011 — Indexed-trader leakage mitigation

**Context.** `OrderPlaced(uint256 indexed batchId, address indexed trader, uint256 orderIndex)` reveals the set of trader addresses participating in each batch. This is a real disclosure documented in `01-architecture.md` leakage map and `07-security.md` Threat 7.

Mitigation candidates:

- **A — Relayer-funded submissions** (Veil-operated meta-tx relayer). Privacy moves from "user wallet → contract" to "relayer wallet → contract"; the relayer correlates out-of-band.
- **B — Stealth addresses (ERC-5564).** Each `placeOrder` from a fresh address derived from the trader's spend key.
- **C — User-side mixer.** Trade from a freshly-mixed address. Out of protocol; user's responsibility.
- **D — Accept and document.** Use a fresh wallet per privacy regime.

**Decision.** Deferred. v1 ships with option D documented. v2 picks one before the Week 5 freeze.

**Consequences.** Until decided, the public set of Veil traders per batch is observable. Any user who cares should use a wallet they have not associated with a public identity.

**Status.** Proposed (pending decision).
